## Task 1: M5 앵커 칸 (Anchor) 구현

### 설명

앵커 칸은 중력을 무시하고 위치가 고정된다. 파괴될 때까지 움직이지 않으며, 해당 열에서 앵커 아래 빈 공간은 버퍼에서 정상 충전된다.

### 상세 구현 노트

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 (앵커는 색 변경을 막지 않음) |
| `can_bfs_traverse()` | `true` | BFS 정상 통과 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 (라인 완성/BFS 그룹 파괴) |
| `on_destroy()` | `{destroyed: true}` | 파괴 시 기믹 제거, 셀 비우기 |
| `on_gravity()` | `false` | **중력 무시** — 낙하하지 않음 |

#### 앵커와 중력의 상세 상호작용

앵커 칸은 S06 Task 5(돌 칸)에서 구현한 **세그먼트 분할 중력** 로직을 공유한다. `on_gravity() → false`인 셀은 돌 칸과 동일하게 열을 세그먼트로 분할한다.

```
예시 (7×1 열, 앵커가 y=3에 위치):

  [빈칸]        [  A  ]    ← 상단 세그먼트 버퍼 충전
  [  A  ]       [  B  ]
  [  B  ]       [  C  ]
  [앵커(빨)]  →  [앵커(빨)]  ← 고정
  [빈칸]        [  D  ]    ← 하단 세그먼트 압축
  [  D  ]       [새 칸]    ← 하단 세그먼트 버퍼 충전
  [빈칸]        [새 칸]
```

#### 앵커 vs 돌 차이점

| 속성 | 앵커 | 돌 |
|------|------|-----|
| 색상 | 있음 (정상 색) | 없음 (-1) |
| BFS | 정상 참여 | 차단 |
| 리컬러 | 가능 | 불가 |
| 파괴 | 가능 | 불가 |
| 중력 | 무시 (고정) | 무시 (고정) |
| 라인 완성 | 참여 (색 판정) | 불가 (색 없음) |

#### 앵커 파괴 후 처리

앵커가 라인 완성으로 파괴되면:
1. 셀이 비워짐
2. 기믹 슬롯 초기화 (NONE)
3. 다음 중력 처리에서 해당 위치는 더 이상 고정 셀이 아님 → 정상 중력 적용
4. 세그먼트가 병합됨

#### 엣지 케이스

1. **앵커 위에 빈 칸**: 앵커 위쪽 세그먼트가 압축/충전된다. 앵커 자체는 움직이지 않으므로 앵커 바로 위에는 빈 칸이 생길 수 없다 (상단 세그먼트가 충전).
2. **앵커 연속 배치**: 같은 열에 앵커가 2개 이상이면 3개 이상 세그먼트로 분할 (돌과 동일한 세그먼트 분할 로직).
3. **앵커 + 돌 같은 열**: 둘 다 `on_gravity()=false`이므로 동일한 세그먼트 분할로 처리. 차이는 앵커는 색이 있고 파괴 가능한 점.
4. **앵커 + 얼음**: 기믹 1개 제한으로 동시 적용 불가.
5. **앵커 리컬러 후 라인 완성**: 앵커 색을 바꿔서 라인을 완성하면 앵커가 파괴되어 중력 구조가 변화한다 → 연쇄 가능성.

#### 비주얼

- 셀 색상 정상 렌더링 + 앵커 아이콘 오버레이
- 앵커 아이콘: ⚓ 형태의 도형 (원 + 아래 갈고리)
- 색상: 셀 색보다 어두운 톤 또는 반투명 흰색/회색
- 구현: `_draw()`에서 `draw_arc()` + `draw_line()` 조합

### 수인 기준

- [ ] 앵커 칸이 중력 처리 시 고정 위치를 유지한다
- [ ] 앵커 위/아래 세그먼트가 독립적으로 압축/충전된다
- [ ] 앵커 칸이 정상적으로 리컬러된다
- [ ] 앵커 칸이 라인 완성 시 파괴되고 세그먼트가 병합된다
- [ ] 앵커 아이콘 오버레이가 셀 위에 표시된다

### 의존성

- S06 Task 1 (기믹 프레임워크)
- S06 Task 5 (돌 칸 — 세그먼트 분할 중력 로직 재사용)
- `scripts/core/gravity.gd` (S06에서 이미 세그먼트 분할 구현됨, 앵커에도 적용 확인)

### 산출물

- `scripts/gimmick/gimmick_anchor.gd`

## Task 2: Gravity 앵커 핸들링 확인 및 보완

### 설명

S06에서 돌 칸을 위해 구현한 세그먼트 분할 중력이 앵커 칸에서도 올바르게 동작하는지 확인하고, 필요 시 보완한다.

### 상세 구현 노트

#### 확인 사항

S06 `gravity.gd`의 세그먼트 분할 로직이 `on_gravity() → false`를 반환하는 **모든** 기믹에 대해 범용적으로 동작하는지 검증:

```
split_column_by_fixed_cells(grid, col_x):
    for y in range(grid.height):
        cell = grid.get_cell(col_x, y)
        handler = GimmickRegistry.get_handler(cell.gimmick_type)
        if not handler.on_gravity(cell, grid):
            # 이 셀 기준으로 세그먼트 분할
            ...
```

이 로직은 돌 칸/앵커 칸을 구분하지 않고 `on_gravity()` 반환값만으로 판단하므로, 앵커가 자동으로 지원된다.

#### 보완이 필요한 경우

1. S06에서 돌 칸을 하드코딩으로 처리했다면 → `on_gravity()` 훅 기반으로 리팩토링
2. 세그먼트 간 버퍼 충전 순서 확인: 상단 세그먼트와 하단 세그먼트 모두 독립적으로 버퍼에서 새 셀을 받아야 한다

#### 앵커 파괴 후 세그먼트 병합 테스트

```
파괴 전:              파괴 후 (앵커 제거 → 중력):
  [  A  ]              [새 칸]
  [  B  ]              [새 칸]
  [앵커(빨)]            [  A  ]
  [  C  ]              [  B  ]
  [  D  ]              [  C  ]
  [  E  ]              [  D  ]
  [  F  ]              [  E  ]
```

앵커가 사라지면 해당 위치가 빈칸이 되고, 다음 중력 처리에서 전체 열이 하나의 세그먼트로 병합되어 정상 압축된다.

### 수인 기준

- [ ] `gravity.gd`가 `on_gravity()` 훅 기반으로 범용적 세그먼트 분할을 수행한다
- [ ] 돌 칸과 앵커 칸이 같은 열에 있을 때 3개 이상 세그먼트로 분할된다
- [ ] 앵커 파괴 후 다음 중력 처리에서 세그먼트가 정상 병합된다
- [ ] 각 세그먼트가 독립적으로 버퍼에서 충전된다

### 의존성

- Task 1 (앵커 구현)
- S06 `scripts/core/gravity.gd`

### 산출물

## Task 3: M6 페인트통 (Paint Bucket) 구현

### 설명

페인트통 칸을 리컬러하면, 해당 칸이 속한 행 전체 또는 열 전체가 같은 색으로 변환된다. 행/열 방향은 셀에 미리 설정되어 있다.

### 상세 구현 노트

#### gimmick_data 활용

```json
{"type": "paint_bucket", "data": {"direction": "row"}}
```

- `direction`: `"row"` 또는 `"col"`

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 (트리거 조건) |
| `can_bfs_traverse()` | `true` | BFS 정상 통과 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 |
| `on_destroy()` | `{destroyed: true}` | 일반 파괴 |
| `on_gravity()` | `true` | 정상 낙하 |
| `on_recolor()` | **행/열 변환 트리거** | 핵심 로직 |

#### `on_recolor()` 상세 로직

```gdscript
func on_recolor(cell, old_color, new_color) -> void:
    var direction = cell.gimmick_data.get("direction", "row")
    var grid = get_grid_reference()  # GameManager 경유 또는 signal

    if direction == "row":
        target_cells = grid.get_row(cell.y)
    else:
        target_cells = grid.get_column(cell.x)

    for target in target_cells:
        if target == cell:
            continue  # 자기 자신은 이미 리컬러됨
        var t_handler = GimmickRegistry.get_handler(target.gimmick_type)
        if t_handler.can_recolor(target, new_color):
            target.color = new_color
            t_handler.on_recolor(target, target.color, new_color)

    # 페인트통 기믹은 1회 사용 후 소모
    cell.clear_gimmick()
```

> **1회용**: 페인트통은 리컬러 1회에 소모된다. 소모 후 일반 셀이 된다.

#### 실행 순서 (중요)

1. 사용자가 셀 터치 → 컬러 큐 색 가져옴
2. BFS 그룹 탐색
3. BFS 그룹 리컬러 수행
4. 그룹 내 페인트통 셀 발견 → `on_recolor()` 호출
5. 행/열 전체 리컬러 수행
6. **페인트통에 의한 리컬러도 다른 기믹의 `can_recolor()`를 존중**
7. 라인 완성 판정 (페인트통으로 행/열이 통일되면 즉시 라인 완성 가능!)

#### 페인트통 연쇄 (중요 엣지 케이스)

페인트통이 행을 리컬러할 때, 해당 행에 또 다른 페인트통이 있으면:
- **연쇄 발동하지 않음**: 페인트통 효과에 의한 리컬러는 `on_recolor()` 재귀 트리거를 하지 않는다.
- 이유: 무한 루프 방지. 구현 시 `is_paint_propagating` 플래그로 제어.

```gdscript
var _is_propagating: bool = false

func on_recolor(cell, old_color, new_color) -> void:
    if _is_propagating:
        return  # 다른 페인트통에 의한 전파 중이면 무시
    _is_propagating = true
    # ... 행/열 변환 로직 ...
    _is_propagating = false
    cell.clear_gimmick()
```

#### 기믹별 페인트통 상호작용

| 대상 기믹 | 페인트통 효과 |
|----------|------------|
| 일반 셀 | 정상 리컬러 |
| 잠긴 칸 (M1) | `can_recolor()=false` → 색 유지 |
| 돌 칸 (M2) | `can_recolor()=false` → 영향 없음 |
| 얼음 칸 (M3) | `can_recolor()=true` → 정상 리컬러 (얼음 유지) |
| 무지개 칸 (M4) | `can_recolor()=true` → 리컬러 + 무지개 소모 |
| 앵커 칸 (M5) | `can_recolor()=true` → 정상 리컬러 |
| 다른 페인트통 (M6) | 연쇄 차단 (위 참조) |
| 코인 칸 (M7) | `can_recolor()=true` → 정상 리컬러 |
| 연쇄 칸 (M8) | `can_recolor()=true` → 정상 리컬러 |

#### 비주얼

- 셀 색상 정상 렌더링 + 페인트통 아이콘 오버레이
- 방향 표시: 가로 화살표(→, row) 또는 세로 화살표(↓, col)
- 구현: 사다리꼴/직사각형(통) + 화살표 `draw_line()` + `draw_polygon()`

### 수인 기준

- [ ] 페인트통 셀 리컬러 시 해당 행(또는 열) 전체가 같은 색으로 변환된다
- [ ] 잠긴/돌 칸은 페인트통 효과에 영향받지 않는다
- [ ] 무지개 칸이 페인트통에 의해 리컬러되면 무지개 기믹이 소모된다
- [ ] 페인트통 연쇄(다른 페인트통 트리거)가 발생하지 않는다
- [ ] 페인트통 사용 후 기믹이 소모되어 일반 셀이 된다
- [ ] 페인트통 효과로 라인이 완성되면 즉시 파괴가 발동한다
- [ ] 방향 표시(행/열)가 비주얼에 표시된다

### 의존성

- S06 기믹 프레임워크
- `scripts/core/flood_fill.gd` (리컬러 후 on_recolor 호출 연결)
- `game_manager.gd` (리컬러 → 페인트통 → 라인 판정 파이프라인)

### 산출물

- `scripts/gimmick/gimmick_paint_bucket.gd`
- `scripts/core/flood_fill.gd` (수정 — on_recolor 호출 추가)
- `game_manager.gd` (수정 — 페인트통 후 추가 라인 판정)

## Task 4: M7 코인 칸 (Coin) 구현

### 설명

코인 칸은 파괴 시 추가 재화를 획득한다. 코인은 셀의 색상/BFS/중력에 영향을 주지 않으며, 순수한 보상 기믹이다.

### 상세 구현 노트

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 |
| `can_bfs_traverse()` | `true` | BFS 정상 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 |
| `on_destroy()` | `{destroyed: true, rewards: {coins: N}}` | 추가 재화 지급 |
| `on_gravity()` | `true` | 정상 낙하 |

#### 보상 처리

`on_destroy()` 반환값의 `rewards` 필드를 `game_manager.gd`(또는 `economy.gd`)에서 수집:

```gdscript
# game_manager.gd — 파괴 처리 후
for cell in destroyed_cells:
    var result = handler.on_destroy(cell)
    if result.has("rewards"):
        economy.add_rewards(result.rewards)
```

#### gimmick_data 활용

```json
{"type": "coin", "data": {"coin_value": 10}}
```

기본값: `coin_value = 10` (JSON에 미지정 시)

#### 비주얼

## Task 5: M8 연쇄 칸 (Chain Multiplier) 구현

### 설명

연쇄 칸이 포함된 파괴에서 파괴 수가 2배로 카운트된다. 스토리 모드의 "N블록 파괴" 목표와 무한 모드의 점수 시스템 모두에 영향을 준다.

### 상세 구현 노트

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 |
| `can_bfs_traverse()` | `true` | BFS 정상 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 |
| `on_destroy()` | `{destroyed: true, effects: [{type: "multiply_count", value: 2}]}` | 파괴 수 배율 |
| `on_gravity()` | `true` | 정상 낙하 |

#### 배율 적용 로직

한 번의 파괴 이벤트(라인 완성 1건)에서 연쇄 칸이 N개 포함되면:
- **방안 (권장)**: 해당 파괴 이벤트의 파괴 카운트를 2배로 계산. 연쇄 칸이 여러 개 있어도 2배 (중첩 불가).
- 이유: 지수적 증가(2^N)는 밸런싱이 어렵다.

```gdscript
func calculate_destroy_count(destroyed_cells: Array) -> int:
    var base_count = destroyed_cells.size()
    var has_chain_mult = false
    for cell in destroyed_cells:
        if cell.gimmick_type == GimmickType.CHAIN_MULT:
            has_chain_mult = true
            break
    return base_count * 2 if has_chain_mult else base_count
```

#### 연쇄 콤보와의 관계

연쇄 콤보(chain combo)의 각 단계에서 독립적으로 연쇄 칸 유무를 판정한다. 1단계 파괴에 연쇄 칸이 없고 2단계 파괴에 있으면, 2단계만 2배.

#### 비주얼

- 셀 색상 정상 렌더링 + "×2" 텍스트 오버레이
- 텍스트: 흰색 + 검정 외곽선, 셀 우하단 배치
- 구현: `draw_string()` 또는 Label 노드

- [ ] 연쇄 칸 포함 파괴 시 파괴 카운트가 2배로 적용된다
- [ ] 연쇄 칸 여러 개가 포함되어도 2배 (중첩 불가)
- [ ] 스토리 모드 목표 진행률과 무한 모드 점수 모두에 배율이 반영된다

## Task 7: S2 번짐 칸 (Spread) — 스토리 모드 전용

### 설명

번짐 칸은 매 턴 종료 시 인접 1칸을 자신의 색으로 감염시킨다. 방치하면 보드 상태가 점점 오염되어 라인 완성 계획이 틀어진다.

### 상세 구현 노트

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 |
| `can_bfs_traverse()` | `true` | BFS 정상 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 |
| `on_destroy()` | `{destroyed: true}` | 일반 파괴 |
| `on_gravity()` | `true` | 정상 낙하 |
| `on_turn()` | **인접 감염 로직** | 핵심 로직 |

#### `on_turn()` 상세 로직

```gdscript
func on_turn(cell, turn_number) -> void:
    var neighbors = grid.get_4_neighbors(cell)
    var valid_targets = []

    for neighbor in neighbors:
        var n_handler = GimmickRegistry.get_handler(neighbor.gimmick_type)
        # 감염 불가 대상 필터링
        if neighbor.gimmick_type == GimmickType.STONE:
            continue  # 돌: 감염 불가
        if neighbor.gimmick_type == GimmickType.LOCKED:
            continue  # 잠긴 칸: 색 변경 불가
        if not n_handler.can_recolor(neighbor, cell.color):
            continue  # 일반적인 can_recolor 체크
        if neighbor.color == cell.color:
            continue  # 이미 같은 색이면 스킵
        valid_targets.append(neighbor)

    if valid_targets.is_empty():
        return  # 감염 가능한 대상 없음

    # 랜덤으로 1칸 선택하여 감염
    var target = valid_targets[randi() % valid_targets.size()]
    target.color = cell.color
```

#### 감염 규칙

1. 4방향(상하좌우) 인접 셀 중 1칸만 감염 (랜덤 선택)
2. 감염 불가 대상: 돌 칸(색 없음), 잠긴 칸(색 변경 불가)
3. 이미 같은 색인 셀은 감염 대상에서 제외
4. 감염 가능한 대상이 없으면 아무 일도 안 함
5. 감염된 셀에는 번짐 기믹이 전파되지 않음 (색만 바뀜)
6. 번짐 칸이 리컬러되면 그 새로운 색으로 다음 턴부터 감염

#### 번짐 + 페인트통 상호작용

- 페인트통이 행/열을 리컬러한 결과 번짐 칸의 색이 바뀔 수 있음
- 번짐 칸의 `on_recolor()`에서 특별한 처리는 없음 — 색만 바뀌고 다음 on_turn에서 새 색으로 감염

#### 번짐 + 무지개 상호작용

- 무지개 칸이 인접해 있으면: 무지개 칸은 `can_recolor()=true`이므로 감염 가능
- 감염되면 무지개가 소모되고 번짐 칸의 색으로 변경됨 (무지개의 `on_recolor()`에서 기믹 제거)

#### 번짐 칸이 여러 개일 때

각 번짐 칸이 독립적으로 `on_turn()` 실행. 실행 순서는 그리드 순회 순서(좌→우, 상→하). 한 번짐 칸의 감염이 다른 번짐 칸의 감염 대상에 영향을 줄 수 있다 (처리 순서 의존).

> **설계 결정**: 순서 의존성을 수용한다. 턴 종료 시 모든 번짐 칸의 감염 대상을 먼저 결정하고 한꺼번에 적용하는 "동시 처리" 방식을 고려할 수도 있으나, 구현 복잡도 대비 게임플레이 영향이 미미하므로 순차 처리로 진행.

#### 비주얼

- 셀 색상 정상 렌더링 + 물방울/번짐 이펙트 오버레이
- 셀 하단에 물방울이 흐르는 듯한 도형 (삼각형 2~3개)
- 색상: 셀 색과 동일 톤, 약간 어두운 반투명
- 구현: `draw_polygon()`으로 물방울 형태

### 수인 기준

- [ ] 매 턴 종료 시 번짐 칸이 인접 1칸을 자기 색으로 감염시킨다
- [ ] 돌 칸, 잠긴 칸은 감염되지 않는다
- [ ] 이미 같은 색인 셀은 감염 대상에서 제외된다
- [ ] 번짐 칸이 파괴되면 더 이상 감염이 발생하지 않는다
- [ ] 번짐 칸 리컬러 후 새 색으로 감염이 진행된다
- [ ] 번짐 비주얼(물방울)이 표시된다

### 의존성

- S06 기믹 프레임워크
- `game_manager.gd` (on_turn 호출 파이프라인, S06 Task 9에서 구현)

## Task 8: S3 퇴색 칸 (Fade) — 스토리 모드 전용

### 설명

퇴색 칸은 리컬러되어도 2턴 후 원래 색으로 되돌아간다. 라인 완성 타이밍을 정확히 맞춰야 하는 퍼즐성을 강화한다.

### 상세 구현 노트

#### gimmick_data 활용

```gdscript
# 초기 상태 (LevelLoader에서 설정)
gimmick_data = {
    "original_color": 2,    # 레벨에서 지정한 원래 색
    "revert_turns": 2,      # 리컬러 후 몇 턴 후 복귀 (기본 2)
    "turn_counter": -1      # -1이면 비활성 (리컬러 안 된 상태)
}
```

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 |
| `can_bfs_traverse()` | `true` | BFS 정상 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 |
| `on_destroy()` | `{destroyed: true}` | 일반 파괴 (기믹 함께 제거) |
| `on_gravity()` | `true` | 정상 낙하 |
| `on_recolor()` | 카운터 시작 | 리컬러 시 타이머 가동 |
| `on_turn()` | 카운터 감소 → 복귀 | 핵심 로직 |

#### `on_recolor()` 상세 로직

```gdscript
func on_recolor(cell, old_color, new_color) -> void:
    if new_color != cell.gimmick_data.original_color:
        # 원래 색과 다른 색으로 칠해지면 카운터 시작/리셋
        cell.gimmick_data.turn_counter = cell.gimmick_data.revert_turns
    else:
        # 원래 색으로 돌아오면 카운터 비활성화
        cell.gimmick_data.turn_counter = -1
```

#### `on_turn()` 상세 로직

```gdscript
func on_turn(cell, turn_number) -> void:
    if cell.gimmick_data.turn_counter <= 0:
        return  # 카운터 비활성 또는 만료 아님

    cell.gimmick_data.turn_counter -= 1

    if cell.gimmick_data.turn_counter == 0:
        # 원래 색으로 복귀!
        cell.color = cell.gimmick_data.original_color
        cell.gimmick_data.turn_counter = -1
        # 복귀 후 라인 완성 체크는 다음 턴에 자연스럽게 처리
```

#### 복귀 후 처리

퇴색 칸이 원래 색으로 돌아갔을 때:
- **라인 완성 재판정**: 원래 색 복귀로 인해 행/열이 완성될 수 있다. 하지만 `on_turn()`은 턴 종료 시점에 호출되므로, 이 시점에서 라인 판정을 다시 수행해야 한다.
- `game_manager.gd`에서 `on_turn()` 완료 후 라인 판정 재실행 필요.

```gdscript
# game_manager.gd
func _on_turn_end(turn_number: int) -> void:
    # 1. 모든 on_turn() 호출
    var state_changed = false
    for cell in grid.all_cells():
        if cell.has_gimmick():
            var handler = GimmickRegistry.get_handler(cell.gimmick_type)
            handler.on_turn(cell, turn_number)
            # 퇴색 등에 의해 색이 변경되었을 수 있음
            state_changed = true

    # 2. 상태 변경 후 라인 판정 재실행
    if state_changed:
        chain_combo.run()  # 연쇄 루프 재실행
```

#### 엣지 케이스

1. **퇴색 칸을 여러 번 리컬러**: 매번 카운터가 리셋된다. 마지막 리컬러 기준으로 2턴 후 복귀.
2. **퇴색 칸을 원래 색으로 리컬러**: 카운터가 비활성화된다 (-1). 이미 원래 색이므로 복귀할 필요 없음.
3. **퇴색 복귀 직후 라인 완성**: 가능하다. 이를 위해 on_turn 후 라인 판정 재실행.
4. **퇴색 + 번짐 인접**: 번짐에 의해 퇴색 칸이 감염되면 → 퇴색의 can_recolor()=true → 감염 적용 → on_recolor()로 카운터 시작 → 2턴 후 원래 색으로 복귀.
5. **퇴색 + 페인트통**: 페인트통에 의해 퇴색 칸이 리컬러되면 → 동일하게 카운터 시작.
6. **퇴색 칸이 파괴 전에 복귀**: 복귀 후에도 기믹은 유지. 다시 리컬러되면 다시 카운터 시작.

#### 비주얼

- 셀 색상 정상 렌더링 + 페이딩 이펙트
- **비활성 상태** (카운터=-1): 셀 색상 위에 미세한 반투명 오버레이 (퇴색 칸임을 표시)
- **카운터 활성 상태**: 셀 색상이 점진적으로 원래 색으로 변화하는 중간색 표현
  - `turn_counter=2`: 현재색 100%
  - `turn_counter=1`: 현재색 50% + 원래색 50% (lerp)
- 구현: 셀 렌더링 시 `Color.lerp()` 적용 + 반투명 ghost 오버레이

- [ ] 복귀 시 라인 완성 판정이 재실행된다

## Task 10: S5 독 칸 (Poison) — 무한 모드 전용

### 설명

독 칸이 필드에 존재하는 동안 시간 감소 속도가 증가한다. 여러 개가 있으면 효과가 누적된다. "빨리 제거해야 하는 긴급 대상" 역할.

### 상세 구현 노트

#### gimmick_data 활용

```json
{"type": "poison", "data": {"speed_multiplier": 1.5}}
```

기본값: `speed_multiplier = 1.5` (시간 감소 속도 1.5배)

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 |
| `can_bfs_traverse()` | `true` | BFS 정상 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 |
| `on_destroy()` | `{destroyed: true}` | 일반 파괴 (제거되면 시간 감소 정상화) |
| `on_gravity()` | `true` | 정상 낙하 |

#### 시간 감소 가속 로직

**액션 기준 틱** (gimmick-candidates.md 명시):
- 독 칸의 시간 감소 가속은 실시간이 아닌 **플레이어 액션 횟수** 기준이다
- 매 액션(리컬러) 시 독 칸 수에 따라 추가 시간이 차감된다

```gdscript
# infinity_mode.gd
func on_action_performed() -> void:
    var poison_count = grid.count_gimmick(GimmickType.POISON)
    if poison_count > 0:
        var total_multiplier = 1.0
        for cell in grid.get_cells_with_gimmick(GimmickType.POISON):
            total_multiplier += (cell.gimmick_data.get("speed_multiplier", 1.5) - 1.0)
        # 추가 시간 차감
        var extra_penalty = BASE_TIME_COST_PER_ACTION * (total_multiplier - 1.0)
        subtract_time(extra_penalty)
```

#### 누적 방식

독 칸이 N개 있을 때:
- 독 칸 1개 (1.5배): 기본 틱 + 기본 틱 × 0.5 = 1.5배 시간 소모
- 독 칸 2개 (각 1.5배): 기본 틱 + 기본 틱 × 0.5 + 기본 틱 × 0.5 = 2.0배
- 독 칸 3개: 2.5배
- **가산 누적** (곱셈 아님) → 예측 가능한 난이도

#### 엣지 케이스

1. **독 칸 파괴 즉시 효과 소멸**: 파괴되는 순간 해당 칸의 배율이 제거됨. 같은 연쇄 내 다음 파괴부터 반영.
2. **독 칸 + 얼음**: 기믹 1개 제한으로 동시 불가.
3. **독 칸 리컬러**: 정상 리컬러 가능. 색을 바꿔도 독 효과 유지.
4. **스토리 모드에서의 독 칸**: 시간 개념이 없으므로 효과 없음. LevelLoader에서 경고.

#### 비주얼

- 셀 색상 정상 렌더링 + 보라색 독 오버레이
- 색상: 보라색 (#9932CC, alpha 0.5) 오버레이
- 셀 모서리에서 독 방울(작은 원) 2~3개 표시
- 구현: `draw_rect()` (반투명 보라) + `draw_circle()` 2~3개

### 수인 기준

- [ ] 독 칸이 존재하는 동안 액션당 시간 감소가 가속된다
- [ ] 독 칸 여러 개의 효과가 가산 누적된다
- [ ] 독 칸 파괴 시 가속 효과가 즉시 소멸된다
- [ ] 스토리 모드에서 독 칸이 있어도 크래시하지 않는다
- [ ] 보라색 독 오버레이가 표시된다
- [ ] `speed_multiplier`가 JSON에서 설정 가능하다

| `"spread"` | 번짐 감염 | 무시 | 카운터 시작 | 기믹 소모 |

```gdscript
# scripts/gimmick/gimmick_rules.gd
class_name GimmickRules

# 페인트통 전파 시 대상 셀 리컬러 가능 여부
static func can_paint_propagate_to(cell) -> bool:
    var handler = GimmickRegistry.get_handler(cell.gimmick_type)
    return handler.can_recolor(cell, -1)  # 색 무관 체크

# 번짐 감염 시 대상 셀 감염 가능 여부
static func can_spread_infect(cell, spread_color: int) -> bool:
    if cell.color == spread_color:
        return false
    var handler = GimmickRegistry.get_handler(cell.gimmick_type)
    return handler.can_recolor(cell, spread_color)

# 라인 완성 판정에서 셀의 색 기여값
static func get_line_color(cell) -> int:
    if cell.gimmick_type == GimmickType.STONE:
        return -99  # 특수값 → 라인 불완성 강제
    var handler = GimmickRegistry.get_handler(cell.gimmick_type)
    if handler.is_bfs_wildcard(cell):
        return -2  # WILDCARD → 아무 색과 매칭
    return cell.color
```

##### 레벨 C: 엣지 케이스 집중 테스트

한 행에 잠긴+무지개+얼음+앵커+페인트통을 집중 배치하여 상호작용 검증.

#### 검증 시나리오

| # | 시나리오 | 검증 포인트 |
|---|---------|------------|
| 1 | 페인트통(행) + 잠긴 칸 같은 행 | 잠긴 칸만 리컬러 차단, 나머지 변환 확인 |
| 2 | 페인트통(행) + 돌 칸 같은 행 | 돌 칸 영향 없음 확인 |
| 3 | 페인트통(행) + 무지개 같은 행 | 무지개 소모 확인 |
| 4 | 페인트통(행) → 행 통일 → 라인 완성 | 즉시 라인 파괴 발동 확인 |
| 5 | 번짐 인접에 잠긴/돌 칸 | 감염 차단 확인 |
| 6 | 번짐 인접에 퇴색 칸 | 감염 + 퇴색 카운터 시작 확인 |
| 7 | 번짐 인접에 무지개 칸 | 무지개 소모 확인 |
| 8 | 퇴색 리컬러 → 2턴 대기 → 복귀 | 원래 색 복귀 확인 |
| 9 | 퇴색 복귀로 라인 완성 | 복귀 후 라인 파괴 발동 확인 |
| 10 | 앵커 + 돌 같은 열 | 3세그먼트 중력 확인 |
| 11 | 앵커 파괴 → 세그먼트 병합 | 중력 구조 변화 확인 |
| 12 | 연쇄 칸 포함 라인 파괴 | 파괴 수 2배 확인 |
| 13 | 코인 칸 파괴 | 재화 추가 확인 |
| 14 | 별 칸 파괴 (스토리) | +1턴 확인 |
| 15 | 시간 칸 파괴 (무한) | +N초 확인 |
| 16 | 독 칸 존재 시 액션 | 시간 가속 확인 |
| 17 | 독 칸 파괴 후 액션 | 가속 해제 확인 |
| 18 | 독 칸 복수 개 | 가산 누적 확인 |
| 19 | 무지개 터치 → 거대 BFS → 라인 완성 → 연쇄 | 전체 파이프라인 확인 |
| 20 | 얼음 crack → 연쇄 중 재완성 → 얼음 파괴 | 연쇄 내 2단계 파괴 확인 |
| 21 | S06 기믹만 있는 레벨 regression | M1~M4 정상 동작 확인 |
| 22 | 기믹 없는 레벨 regression | 기존 챕터 1 레벨 정상 동작 확인 |
| 23 | 페인트통 연쇄 차단 | 같은 행에 페인트통 2개 → 연쇄 안 함 확인 |
| 24 | 번짐 → 페인트통 감염 | 색만 변경, 행/열 변환 트리거 안 함 확인 |

#### 코어 루프 최종 파이프라인 (기믹 통합)

```
1. 사용자 터치 → 컬러 큐 색 확보
2. BFS 그룹 탐색
   - can_bfs_traverse() 체크 (돌: 차단)
   - is_bfs_wildcard() 체크 (무지개: 와일드)
3. BFS 그룹 리컬러
   - can_recolor() 체크 (잠긴: 차단)
   - on_recolor() 호출 (페인트통: 행/열 변환, 무지개: 기믹 소모, 퇴색: 카운터 시작)
4. 페인트통 행/열 변환 (BFS 리컬러의 부수 효과)
   - 변환 대상 셀에 can_recolor() 체크
   - on_recolor(source="paint") 호출
5. 라인 완성 판정
   - 돌 칸 포함 → 불완성
   - 무지개 → 와일드카드
   - get_line_color() 사용
6. 라인 파괴 처리
   - can_destroy() 체크 (돌: 불가)
   - on_destroy() 호출 (얼음: 내구도 감소, 코인: 재화, 연쇄: 2배, 별: +턴, 시간: +초)
7. 중력 처리
   - on_gravity() 체크 (돌/앵커: 고정)
   - 세그먼트 분할 압축
8. 연쇄 루프 (5~7 반복, MAX_CHAIN=20)
9. 턴 종료
   - 독 칸 존재 시 시간 가속 적용 (무한 모드)
   - on_turn() 호출 (번짐: 감염, 퇴색: 카운터)
   - on_turn 후 상태 변경 감지 → 라인 판정 재실행 가능
10. 턴 카운트/타이머 갱신
```
