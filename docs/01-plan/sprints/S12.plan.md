# S12: 시뮬레이터 + 에디터 + 100스테이지 (Week 12)

> **Milestone**: M6 — Tools & Levels
> **스프린트 목표**: 밸런싱 도구 체인(시뮬레이터 + 에디터 + 자동 생성기)을 완성하고, 나머지 90스테이지 레벨 + 스토리 데이터를 확정하여 모든 기능 개발을 완료한다
> **선행 조건**: S11 완료 (모든 코어 시스템, UI, 기믹, 스토리, 이마젠, 컬렉션, 수익화 구현 완료)
> **중요**: 이 스프린트가 기능 개발의 마지막. S13부터는 폴리싱/QA만 진행.

## 설계 개요

### 도구 체인 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│                  Level Production Pipeline                │
│                                                           │
│  ┌──────────────┐    ┌──────────────┐    ┌─────────────┐ │
│  │ Level Editor  │───▶│  JSON Export  │───▶│  Simulator  │ │
│  │  (수동 편집)   │    │  (레벨 데이터)  │    │  (검증)     │ │
│  └──────────────┘    └──────────────┘    └─────────────┘ │
│         ▲                    ▲                    │       │
│         │                    │                    ▼       │
│         │            ┌──────────────┐    ┌─────────────┐ │
│         └────────────│ Auto-Generator│◀───│  Validation │ │
│      (수동 보정)      │  (자동 생성)   │    │  Pipeline   │ │
│                      └──────────────┘    │  (합격/불합격) │ │
│                                          └─────────────┘ │
└─────────────────────────────────────────────────────────┘

실행 방식:
  1. Auto-Generator로 레벨 일괄 생성
  2. Simulator로 N회 시뮬레이션 → 통계 산출
  3. Validation Pipeline으로 합격/불합격 자동 판정
  4. 불합격 레벨 → Level Editor에서 수동 보정
  5. 보정 후 재검증 → 최종 확정
```

### 시뮬레이터 설계 원칙

```
핵심 원칙: UI 무관, 데이터 레이어만 사용

시뮬레이터는 다음만 사용:
  - Grid (데이터 모델)
  - FloodFill (BFS 탐색)
  - RowDestroy (라인 파괴 판정)
  - Gravity (중력 낙하)
  - ChainCombo (연쇄 루프)
  - ColorQueue (큐 스케줄링)
  - TurnManager (턴 관리)
  - GimmickManager (기믹 처리)

시뮬레이터가 사용하지 않는 것:
  - 어떤 Node2D/Control도 사용하지 않음
  - 렌더링, 애니메이션, 사운드 없음
  - Tween, Timer 없음 (모든 것이 즉시 실행)
  - 입력 시스템 없음 (AI 전략이 액션 결정)
```

### 챕터별 기믹 롤아웃 매트릭스

```
Ch  | 새 기믹             | 누적 기믹
----|--------------------|---------------------------------
 1  | (없음)              | 없음
 2  | 얼음 칸 (Ice)       | 얼음
 3  | 잠금 칸 (Lock)      | 얼음, 잠금
 4  | 무지개 칸 (Rainbow)  | 얼음, 잠금, 무지개

 6  | 페인트통 (Paint)     | 얼음, 잠금, 무지개, 돌, 페인트통
 7  | 앵커 (Anchor)       | 얼음, 잠금, 무지개, 돌, 페인트통, 앵커
```

## Task 1: 밸런싱 시뮬레이터 코어

### 설명

게임의 데이터 레이어만 사용하여 UI 렌더링 없이 게임 로직을 실행하는 headless 시뮬레이터. AI 전략이 매 턴 액션을 결정하고, 게임이 종료(클리어/실패)될 때까지 루프를 실행한다.

### 수용 기준 (Acceptance Criteria)

- [ ] `Simulator` 클래스: `RefCounted` 상속 (Node 아닌 순수 데이터 객체)
- [ ] `simulate(level_data: Dictionary, strategy: AIStrategy, seed: int = -1) -> SimulationResult`
- [ ] `SimulationResult` 구조:
  ```
  {
    "cleared": bool,
    "turns_used": int,
    "turns_remaining": int,
    "total_destroyed": int,
    "max_chain": int,
    "stars": int,        # 스토리 모드 기준
    "score": int,        # 무한 모드 기준
    "actions": Array     # 각 턴의 액션 로그
  }
  ```
- [ ] 게임 로직 실행 순서 (매 턴):
  1. AI 전략이 (row, col) 액션 결정
  2. BFS 리컬러
  3. 라인 파괴 판정
  4. 파괴 실행 + 통계 누적
  5. 중력 낙하
  6. 연쇄 콤보 루프 (파괴→중력→재판정 반복)
  7. 턴 관리 업데이트
  8. 기믹 턴 후 처리 (번짐, 퇴색, 독 등)
  9. 게임 종료 조건 체크
- [ ] 기믹 지원: GimmickManager의 데이터 레이어 메서드 호출 (UI 관련 메서드 제외)
- [ ] 랜덤 시드 지원: seed 지정 시 재현 가능한 결과
- [ ] 스토리/무한 모드 모두 지원: level_data의 mode 필드로 구분
- [ ] MAX_TURNS 안전장치: 300턴 초과 시 강제 종료 (무한 루프 방지)
- [ ] 배치 실행: `simulate_batch(level_data, strategy, n_runs, seed_base) -> Array[SimulationResult]`

### 의존성

- `Grid` (S01) — 그리드 데이터 모델
- `FloodFill` (S02) — BFS 탐색
- `RowDestroy` (S02) — 라인 파괴 판정
- `Gravity` (S02) — 중력 낙하
- `ChainCombo` (S03) — 연쇄 루프
- `ColorQueue` (S03) — 큐 스케줄링
- `TurnManager` (S03) — 턴 관리
- `GimmickManager` (S06) — 기믹 처리

### 주요 구현 노트

- **핵심: Node 의존성 제거**. 기존 코어 로직이 Node를 상속하거나 씬 트리에 의존하는 경우, 데이터 레이어를 분리하거나 시뮬레이터 전용 래퍼를 작성해야 함
- 기존 `grid.gd`, `flood_fill.gd` 등이 Node 상속인 경우:
  - 옵션 A: 로직을 RefCounted 기반 클래스로 리팩터 (권장)
  - 옵션 B: 시뮬레이터에서 SceneTree 없이 Node를 인스턴스화하여 사용 (Godot 4.x에서 가능하나 제한적)
  - 옵션 C: 시뮬레이터 전용 경량 로직 복제 (최후 수단)
- 시뮬레이터는 Godot 에디터 내에서 EditorScript로 실행하거나, `--headless` 모드로 커맨드라인 실행
- 배치 실행 시 멀티스레드 지원 고려: `Thread` 클래스로 병렬 시뮬레이션 (선택적 최적화)
- 액션 로그는 디버깅 용도: `[{"turn": 1, "action": [3, 4], "color": 2, "destroyed": 5, "chain": 1}]`

### 산출물 파일 경로

- `tools/simulator/simulator.gd`
- `tools/simulator/simulation_result.gd`

## Task 2: AI 전략 구현

### 설명

시뮬레이터에서 매 턴 어떤 셀을 터치할지 결정하는 AI 전략. 3가지 전략(Greedy, Random, Semi-random)을 구현하고, 공통 인터페이스로 교체 가능하게 한다.

### 수용 기준 (Acceptance Criteria)

- [ ] `AIStrategy` 기본 클래스 (RefCounted):
  ```gdscript
  func decide_action(grid: GridData, color_queue: ColorQueueData, turn_info: Dictionary) -> Vector2i
  ```
- [ ] **GreedyStrategy**:
  - 모든 가능한 셀을 순회
  - 각 셀에 대해 "이 셀을 터치했을 때 완성되는 라인(행/열) 수"를 평가
  - 라인 완성 수가 가장 많은 셀 선택
  - 라인 완성이 없는 경우: BFS 그룹 크기가 가장 큰 셀 선택 (큰 그룹 리컬러 → 향후 라인 완성 확률 증가)
  - 동점 시: 랜덤 선택
- [ ] **RandomStrategy**:
  - 현재 큐 색과 다른 색을 가진 셀 중 랜덤 선택
  - 유효하지 않은 셀(같은 색) 필터링
- [ ] **SemiRandomStrategy**:
  - 70% 확률로 Greedy 선택
  - 30% 확률로 Random 선택
  - "평균적인 플레이어"를 모사
- [ ] 각 전략의 `decide_action()`이 유효한 좌표를 반환 (그리드 범위 내, 큐 색과 다른 색)
- [ ] 유효한 액션이 없는 경우 (모든 셀이 큐 색과 같음): `Vector2i(-1, -1)` 반환 (턴 스킵)
- [ ] 전략별 평균 실행 시간: 7x7 그리드 기준 1ms 이하

### 의존성

- 시뮬레이터 코어 (Task 1) — 전략을 주입받아 실행
- `Grid`, `FloodFill`, `RowDestroy` — 평가 함수에서 사용

### 주요 구현 노트

- Greedy 평가 함수:
  ```gdscript
  func _evaluate_cell(row: int, col: int, grid: GridData, queue_color: int) -> float:
      # 1. 이 셀의 BFS 그룹을 queue_color로 변경했을 때의 가상 그리드 생성
      # 2. 가상 그리드에서 완성되는 라인 수 카운트
      # 3. score = line_count * 100 + bfs_group_size
      return score
  ```
- 가상 그리드는 원본을 복사(deep copy)하여 사용 — 원본 그리드 변경 금지
- Grid의 `duplicate()` 또는 `clone()` 메서드가 필요할 수 있음 (없으면 추가)
- SemiRandomStrategy의 70/30 비율은 `@export`로 조정 가능하게
- 향후 추가 가능 전략: LookAhead (2턴 앞 시뮬레이션), MaxDestruction (최대 파괴 블록 수 우선)

### 산출물 파일 경로

- `tools/simulator/ai_strategy.gd` (기본 클래스)
- `tools/simulator/greedy_strategy.gd`
- `tools/simulator/random_strategy.gd`
- `tools/simulator/semi_random_strategy.gd`

## Task 3: 시뮬레이션 통계 출력

### 설명

N회 시뮬레이션 결과를 집계하여 클리어율, 평균 턴, 평균 파괴 블록 수, 별 분포 등의 통계 리포트를 생성한다. 콘솔 출력 + JSON 파일 내보내기를 지원한다.

### 수용 기준 (Acceptance Criteria)

- [ ] `SimulationStats` 클래스:
  ```gdscript
  func aggregate(results: Array[SimulationResult]) -> Dictionary
  ```
- [ ] 산출 통계 항목:
  - `total_runs`: 총 시뮬레이션 횟수
  - `clear_rate`: 클리어율 (0.0 ~ 1.0)
  - `avg_turns_used`: 평균 사용 턴 수
  - `avg_turns_remaining`: 평균 남은 턴 수 (클리어 시)
  - `avg_destroyed`: 평균 파괴 블록 수
  - `avg_max_chain`: 평균 최대 연쇄 수
  - `star_distribution`: 별 분포 `{1: N, 2: N, 3: N, 0: N(실패)}`
  - `min_turns` / `max_turns`: 최소/최대 사용 턴
  - `std_dev_turns`: 턴 수 표준편차
  - `score_avg` / `score_max`: 점수 평균/최대 (무한 모드)
- [ ] 콘솔 출력 포맷:
  ```
  === Simulation Report ===
  Level: chapter_02_stage_01
  Strategy: SemiRandom
  Runs: 1000
  ---
  Clear Rate: 72.3%
  Avg Turns Used: 18.4 / 25
  Avg Destroyed: 67.2
  Avg Max Chain: 3.1
  Star Distribution: ★★★ 23.1% | ★★ 31.5% | ★ 17.7% | Fail 27.7%
  Turn Range: 8 ~ 25 (σ = 4.2)
  ---
  Verdict: PASS (target: 60-80%)
  ```
- [ ] JSON 리포트 내보내기: `tools/reports/` 디렉토리에 타임스탬프 파일명으로 저장
- [ ] 복수 레벨 일괄 리포트: 여러 레벨의 통계를 하나의 요약 테이블로 출력
- [ ] 난이도 곡선 시각화 데이터: 챕터/스테이지 순서대로 클리어율 배열 → JSON으로 내보내기 (외부 도구에서 차트 생성 가능)

### 의존성

- 시뮬레이터 코어 (Task 1) — SimulationResult 배열 입력
- AI 전략 (Task 2) — 전략별 통계 비교

### 주요 구현 노트

- 표준편차 계산: `sqrt(sum((x - mean)^2) / n)`
- JSON 리포트 구조:
  ```json
  {
    "timestamp": "2026-04-10T14:30:00",
    "level_id": "chapter_02_stage_01",
    "strategy": "SemiRandom",
    "total_runs": 1000,
    "clear_rate": 0.723,
    "star_distribution": {"3": 231, "2": 315, "1": 177, "0": 277},
    ...
  }
  ```
- 일괄 리포트는 CSV 형식도 지원 (스프레드시트에서 분석 편의)
- 리포트 디렉토리가 없으면 자동 생성 (`DirAccess.make_dir_recursive_absolute()`)

### 산출물 파일 경로

- `tools/simulator/simulation_stats.gd`
- `tools/reports/` (리포트 출력 디렉토리)

## Task 4: 레벨 에디터 UI

### 설명

Godot 에디터 내에서 또는 별도 씬으로 실행되는 레벨 에디터. 그리드 크기를 설정하고, 각 셀의 색상과 기믹을 배치하고, 프리뷰 플레이를 실행하고, JSON으로 내보낼 수 있다.

### 수용 기준 (Acceptance Criteria)

- [ ] 에디터 화면 레이아웃:
  - 좌측: 도구 팔레트 (색상 선택, 기믹 선택, 지우기)
  - 중앙: 그리드 편집 영역 (메인 + 버퍼)
  - 우측: 레벨 속성 패널 (그리드 크기, 턴 수, 목표 블록 수, 별 기준, 모드)
  - 하단: 버튼 바 (`[새 레벨]` `[불러오기]` `[저장]` `[JSON 내보내기]` `[프리뷰]` `[시뮬레이션]`)
- [ ] 그리드 크기 설정: 5x5, 6x6, 7x7 선택 (드롭다운)
- [ ] 셀 색상 배치: 팔레트에서 색 선택 → 셀 클릭하여 색 지정
- [ ] 기믹 배치: 팔레트에서 기믹 선택 → 셀 클릭하여 기믹 추가
  - 기믹 종류: 얼음, 잠금, 무지개, 돌, 페인트통, 앵커, 번짐, 퇴색, 시간, 독, 더블, 폭탄, 블랙홀
  - 셀당 1개 기믹 (기존 기믹 덮어쓰기)
- [ ] 기믹 제거: 지우기 도구로 셀 클릭 시 기믹 제거 (색상은 유지)
- [ ] 레벨 속성 편집:
  - 모드: story / infinity
  - 턴 제한 (스토리): SpinBox (10 ~ 50)
  - 목표 파괴 블록 수 (스토리): SpinBox
  - 시간 제한 (무한): SpinBox (초)

## Task 5: 레벨 자동 생성기

### 설명

파라미터(그리드 크기, 색상 수, 기믹 종류/밀도, 목표, 턴 수)를 입력받아 레벨 JSON을 자동 생성하는 도구. 챕터별 기믹 롤아웃 매트릭스에 따라 적절한 기믹을 배치한다.

### 수용 기준 (Acceptance Criteria)

- [ ] `LevelGenerator` 클래스 (RefCounted):
  ```gdscript
  func generate(params: LevelGenParams) -> Dictionary  # 레벨 JSON
  ```
- [ ] `LevelGenParams` 구조:
  ```
  {
    "grid_size": int,           # 5, 6, 7
    "num_colors": int,          # 3 ~ 6
    "mode": String,             # "story" / "infinity"
    "turn_limit": int,          # 스토리 모드 턴 제한
    "destroy_goal": int,        # 스토리 모드 목표 블록 수
    "time_limit": float,        # 무한 모드 시간 제한
    "star_thresholds": Array,   # [3성 기준, 2성 기준, 1성 기준]
    "gimmick_types": Array,     # 사용할 기믹 종류 ["ice", "lock", ...]
    "gimmick_density": float,   # 0.0 ~ 0.5 (기믹 셀 비율)
    "difficulty": String,       # "easy", "normal", "hard"
    "seed": int                 # 랜덤 시드 (-1 = 자동)
  }
  ```
- [ ] 그리드 생성 알고리즘:
  1. 메인 그리드 + 버퍼를 num_colors 색상으로 랜덤 채움
  2. 초기 상태에서 완성된 라인이 없도록 보장 (`ensure_no_completed_lines`)
  3. 기믹 배치: gimmick_density 비율만큼 랜덤 셀에 기믹 할당
  4. 기믹 배치 규칙: 돌 칸은 라인 중앙 배치 금지, 무지개는 기믹 중첩 금지 등
- [ ] 난이도 프리셋:
  - `easy`: 색 3~4개, 넉넉한 턴, 기믹 밀도 0.0~0.1
  - `normal`: 색 4~5개, 보통 턴, 기믹 밀도 0.1~0.2
  - `hard`: 색 5~6개, 빠듯한 턴, 기믹 밀도 0.2~0.4
- [ ] 챕터 프리셋: `generate_for_chapter(chapter: int, stage_in_chapter: int) -> Dictionary`
  - 챕터별 기믹 롤아웃 매트릭스 자동 적용
  - 스테이지 번호에 따라 난이도 점진적 증가 (1~3: easy, 4~7: normal, 8~10: hard)
- [ ] 배치 생성: `generate_chapter(chapter: int, count: int = 10) -> Array[Dictionary]`
- [ ] 생성된 레벨 JSON은 기존 레벨 데이터 스키마와 100% 호환

### 의존성

- 레벨 데이터 스키마 (S04) — JSON 형식 준수
- 기믹 데이터 (S06~S07) — 기믹 종류 및 규칙
- 챕터별 기믹 롤아웃 (Task 7) — 매트릭스 데이터

### 주요 구현 노트

- `ensure_no_completed_lines`: 생성 후 완성 라인 발견 시 해당 라인의 랜덤 1~2셀 색상 변경하여 해소
- 기믹 배치 제약 조건:
  - 돌(Stone): 메인 그리드의 행/열 중앙 3칸에는 배치 금지 (라인 완성 불가능 방지)
  - 잠금(Lock): 같은 행/열에 3개 이상 배치 금지
  - 앵커(Anchor): 최하단 행에만 배치 (중력 의미 있는 위치)
  - 무지개(Rainbow): 기존 기믹과 중첩 금지
- 턴 수 자동 계산 옵션: turn_limit = -1이면 "셀 수 / 2.5 + 기믹 보정" 공식으로 자동 산출
- 목표 블록 수 자동 계산: destroy_goal = -1이면 "전체 셀 × 0.6" 공식으로 자동 산출
- 별 기준 자동 계산: 3성 = 남은 턴 ≥ 40%, 2성 = 남은 턴 ≥ 20%, 1성 = 클리어

## Task 6: 난이도 검증 파이프라인

### 설명

자동 생성된 레벨을 시뮬레이터로 N회 실행하여 클리어율이 목표 범위(60~80%) 내에 있는지 자동 판정하는 파이프라인. 불합격 레벨은 파라미터를 자동 조정하여 재생성을 시도한다.

### 수용 기준 (Acceptance Criteria)

- [ ] `ValidationPipeline` 클래스:
  ```gdscript
  func validate(level_data: Dictionary, n_runs: int = 100) -> ValidationResult
  func validate_and_fix(level_data: Dictionary, gen_params: LevelGenParams, max_retries: int = 5) -> Dictionary
  ```
- [ ] `ValidationResult` 구조:
  ```
  {
    "passed": bool,
    "clear_rate": float,
    "target_min": float,       # 0.6
    "target_max": float,       # 0.8
    "stats": Dictionary,       # SimulationStats 전체
    "verdict": String,         # "PASS", "TOO_EASY", "TOO_HARD"
    "suggestion": String       # 조정 제안 (예: "턴 수 -2 권장")
  }
  ```
- [ ] 판정 기준:
  - PASS: 60% ≤ clear_rate ≤ 80%
  - TOO_EASY: clear_rate > 80% → 턴 감소 / 기믹 추가 / 목표 증가 제안
  - TOO_HARD: clear_rate < 60% → 턴 증가 / 기믹 제거 / 색상 수 감소 제안
- [ ] 자동 조정 (`validate_and_fix`):
  - TOO_EASY → 턴 -1~2, 기믹 밀도 +0.05 → 재생성 → 재검증
  - TOO_HARD → 턴 +1~2, 기믹 밀도 -0.05, 색상 -1 → 재생성 → 재검증
  - max_retries 횟수까지 반복, 그래도 실패 시 가장 가까운 결과 반환 + 경고
- [ ] 시뮬레이션 전략: SemiRandom 기본 (평균 플레이어 모사)
- [ ] 배치 검증: `validate_batch(levels: Array, n_runs: int) -> Array[ValidationResult]`
- [ ] 진행 상황 시그널: `signal validation_progress(current: int, total: int)`

### 의존성

- 시뮬레이터 (Task 1) — 시뮬레이션 실행
- AI 전략 (Task 2) — SemiRandom 전략
- 통계 모듈 (Task 3) — 결과 집계
- 레벨 생성기 (Task 5) — 재생성

### 주요 구현 노트

- 자동 조정의 턴 수 변경 폭: `adjustment = int((target_rate - actual_rate) * 10)` (비례 조정)
- 기믹 밀도 조정 폭: `±0.05` (미세 조정)
- 색상 수 조정: 최소 3, 최대 6으로 클램핑
- 검증 실행 시간 목표: 레벨 1개당 100회 시뮬 → 5초 이내 (7x7 그리드 기준)
- 배치 검증 시 전체 소요 시간 표시: `"검증 완료: 100/100 레벨, 소요 시간: 8분 32초"`
- 자동 조정 이력 기록: 어떤 파라미터를 어떻게 변경했는지 로그

### 산출물 파일 경로

- `tools/simulator/validation_pipeline.gd`
- `tools/simulator/validation_result.gd`

  - 전체: Ch1(가장 쉬움) → Ch10(가장 어려움) 점진 상승
  - 급격한 난이도 변화 없음 (인접 스테이지 간 클리어율 차이 ≤ 15%)
- [ ] 별 분포 검증:

  - 1성(겨우 클리어): 10~25%
  - 실패: 20~40%
- [ ] Greedy 전략 검증: 클리어율 85~95% (상한 검증 — 최적 플레이 시 거의 클리어 가능)
- [ ] Random 전략 검증: 클리어율 10~30% (하한 검증 — 순수 운으로는 어려움)
- [ ] 종합 리포트 JSON 생성: 100스테이지 전체 통계 + 난이도 곡선 데이터
- [ ] 불합격 스테이지 목록 출력: 조건 미충족 스테이지와 문제점 명시

## Task 10: 챕터 2~10 스토리 스크립트

### 설명

챕터 2~10의 90스테이지에 대한 전/후 다이얼로그 스크립트를 작성한다. 각 챕터의 스토리 아크, 등장 캐릭터(후냐 + 이마젠), 분위기를 설정하고, 스토리 데이터 스키마(S08)에 맞는 JSON으로 작성한다.

### 수용 기준 (Acceptance Criteria)

- [ ] 챕터 2~10, 각 10스테이지 = 총 90스테이지의 전/후 대사
- [ ] 각 스테이지: pre_dialogue (스테이지 시작 전) + post_dialogue (클리어 후)
- [ ] 대사 분량 가이드:
  - 일반 스테이지: pre 2~4줄, post 1~3줄
  - 챕터 시작 (Stage 1): pre 5~8줄 (챕터 도입)
  - 챕터 끝 (Stage 10): post 5~8줄 (챕터 마무리 + 다음 떡밥)
  - 이마젠 합류 스테이지: post에 합류 이벤트 대사 추가
- [ ] 스토리 아크 구조 (전체):
  - Ch1: 색을 잃은 마을 발견 (도입) — 이미 작성됨 (S08)
  - Ch2: 숲 지역 — 얼음에 갇힌 이마젠 구출
  - Ch3: 동굴 — 잠겨진 문 뒤의 비밀
  - Ch4: 호수 — 무지개 빛 호수의 이마젠
  - Ch5: 산 — 돌 장벽 너머의 마을
  - Ch6: 마을 — 페인트 장인 이마젠
  - Ch7: 절벽 — 앵커로 고정된 유적
  - Ch8: 늪 — 번져가는 어둠과의 싸움
  - Ch9: 탑 — 퇴색하는 기억의 장소
  - Ch10: 성 — 최종 대결, 모든 색 되찾기
- [ ] 각 챕터에서 1마리 이마젠 합류 (Ch1 포함 총 10마리)
- [ ] 후냐의 성격 일관성: 밝고 호기심 많은, 이마젠을 좋아하는 캐릭터
- [ ] S08 스토리 데이터 스키마 준수:
  ```json
  {
    "speaker": "후냐",
    "text": "대사 내용",
    "position": "left",
    "expression": "happy",
    "event": null
  }
  ```
- [ ] 이마젠 합류 이벤트 트리거: `"event": "imagen_join:imagen_name"`
- [ ] 챕터 타이틀 데이터: 각 챕터의 타이틀 텍스트와 부제

### 의존성

- 스토리 데이터 스키마 (S08) — JSON 형식 준수
- 이마젠 데이터 (S09) — 이마젠 이름, 성격 참조
- 챕터 1 스크립트 (S08) — 기존 스토리와 연속성

### 주요 구현 노트

- 전체 아크: "색을 잃은 세계에서 후냐가 이마젠들과 함께 색을 되찾는 여정"
