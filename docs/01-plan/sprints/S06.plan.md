# S06: 기믹 프레임워크 + 공용 기믹 핵심 4종

> **Milestone**: M3 — Gimmick
> **Sprint 목표**: 기믹 시스템의 기반 아키텍처를 설계/구현하고, 코어 루프에 가장 깊이 관여하는 공용 기믹 4종(잠긴/돌/얼음/무지개)을 완성한다. 이후 S07에서 나머지 9종을 추가할 때 프레임워크 변경이 최소화되도록 확장성을 확보한다.

---

## 완료 기준 (Sprint-level Acceptance Criteria)

1. 기믹이 배치된 레벨 JSON을 로드하여 4종 기믹이 그리드에 정상 표시된다
2. 잠긴 칸: BFS 리컬러 불가, 라인 완성 시 파괴 확인
3. 돌 칸: BFS 차단, 파괴 불가, 중력 무시(고정) 확인
4. 얼음 칸: 1회 라인 파괴 → 금(crack), 2회 → 완전 파괴 확인
5. 무지개 칸: 어떤 색과도 BFS 연결, 리컬러 시 지정 색으로 변경 확인
6. 4종 기믹이 동시 배치된 테스트 레벨에서 코어 루프(리컬러→라인 파괴→중력→연쇄)가 정상 동작
7. 기존 기믹 없는 레벨(챕터 1 등)에서 regression 없음

## Task 1: 기믹 베이스 클래스 (`gimmick_base.gd`)

### 설명

모든 기믹이 상속하는 추상 베이스 클래스를 정의한다. 기믹 타입 enum, 라이프사이클 훅 인터페이스, 공통 유틸리티를 제공한다.

### 상세 구현 노트

#### GimmickType enum 정의

```
enum GimmickType {
    NONE = 0,
    # 공용 (M 시리즈)
    LOCKED = 1,        # M1 잠긴 칸
    STONE = 2,         # M2 돌 칸
    ICE = 3,           # M3 얼음 칸
    RAINBOW = 4,       # M4 무지개 칸
    ANCHOR = 5,        # M5 앵커 칸
    PAINT_BUCKET = 6,  # M6 페인트통
    COIN = 7,          # M7 코인 칸
    CHAIN_MULT = 8,    # M8 연쇄 칸
    # 스토리 전용 (S 시리즈)
    STAR = 10,         # S1 별 칸
    SPREAD = 11,       # S2 번짐 칸
    FADE = 12,         # S3 퇴색 칸
    # 무한 전용
    TIME = 20,         # S4 시간 칸
    POISON = 21,       # S5 독 칸
}
```

> 번호 사이에 gap을 두어 향후 기믹 추가 여지를 남긴다.

#### 라이프사이클 훅 (virtual methods)

| 훅 | 호출 시점 | 반환 | 설명 |
|---|----------|------|------|
| `can_recolor(cell, new_color) → bool` | BFS 리컬러 직전 | `true`=허용 / `false`=차단 | 잠긴 칸: false, 돌: false |
| `on_recolor(cell, old_color, new_color)` | 리컬러 완료 직후 | void | 페인트통: 행/열 전환 트리거 |
| `can_bfs_traverse(cell, from_cell) → bool` | BFS 탐색 중 인접 셀 진입 판단 | `true`=통과 / `false`=차단 | 돌: false |
| `is_bfs_wildcard(cell) → bool` | BFS 색 비교 시 | `true`=모든 색 매칭 | 무지개: true |
| `can_destroy(cell) → bool` | 라인 파괴 대상 판정 | `true`=파괴 가능 / `false`=불가 | 돌: false |
| `on_destroy(cell) → Dictionary` | 파괴 처리 직후 | `{destroyed: bool, rewards: {}, effects: []}` | 얼음: 1회차면 crack만 적용, destroyed=false |
| `on_gravity(cell, grid) → bool` | 중력 낙하 판정 | `true`=정상 낙하 / `false`=고정 | 앵커: false |
| `on_turn(cell, turn_number)` | 턴 종료 시 | void | 번짐: 인접 감염, 퇴색: 카운터 체크 |
| `get_visual_config(cell) → Dictionary` | 렌더링 시 | `{overlay, tint, shape, ...}` | 각 기믹별 비주얼 설정 반환 |

#### 기믹 레지스트리

`GimmickRegistry` (autoload 또는 static class)를 만들어 `GimmickType → GimmickHandler` 매핑을 관리한다:

```gdscript
# gimmick_registry.gd
static var handlers: Dictionary = {}

static func register(type: GimmickType, handler: GimmickBase) -> void
static func get_handler(type: GimmickType) -> GimmickBase
static func has_handler(type: GimmickType) -> bool
```

이렇게 하면 코어 로직(flood_fill, row_destroy, gravity 등)이 개별 기믹 클래스를 직접 참조하지 않고 레지스트리를 통해 훅을 호출할 수 있다.

### 수인 기준 (Acceptance Criteria)

- [ ] `GimmickType` enum이 13종 + NONE을 포함한다
- [ ] 모든 라이프사이클 훅이 virtual method로 선언되어 있다 (기본 구현은 "아무것도 안 함" 또는 true 반환)
- [ ] `GimmickRegistry`에 핸들러를 등록/조회할 수 있다
- [ ] 기존 코어 스크립트가 기믹 미설정(NONE) 셀에서도 동일하게 동작한다

### 의존성

- 없음 (신규 파일)

### 산출물

- `scripts/gimmick/gimmick_base.gd`
- `scripts/gimmick/gimmick_registry.gd`

## Task 2: Cell 기믹 슬롯 확장

### 설명

`cell.gd`에 기믹 관련 필드를 추가한다. 하나의 셀은 최대 1개의 기믹을 가질 수 있다.

### 상세 구현 노트

#### 추가 필드

```gdscript
# cell.gd 기존 필드에 추가
var gimmick_type: int = GimmickType.NONE   # 기믹 타입
var gimmick_state: int = 0                 # 기믹별 내부 상태 (예: 얼음의 crack 여부)
var gimmick_durability: int = 0            # 내구도 (예: 얼음 = 2)
var gimmick_data: Dictionary = {}          # 기믹별 추가 데이터 (예: 페인트통 방향, 퇴색 원래색, 시간칸 보너스 초)
```

#### gimmick_data 용도 예시

| 기믹 | gimmick_data 키 | 설명 |
|------|-----------------|------|
| M6 페인트통 | `direction: "row" \| "col"` | 행/열 방향 |
| S3 퇴색 칸 | `original_color: int, turn_counter: int` | 원래 색과 리컬러 후 경과 턴 |
| S4 시간 칸 | `bonus_seconds: float` | 파괴 시 추가 시간 |
| S5 독 칸 | `speed_multiplier: float` | 시간 감소 가속 배율 |

#### 유틸리티 메서드 추가

```gdscript
func has_gimmick() -> bool:
    return gimmick_type != GimmickType.NONE

func set_gimmick(type: int, durability: int = 0, data: Dictionary = {}) -> void:
    gimmick_type = type
    gimmick_durability = durability
    gimmick_state = 0
    gimmick_data = data

func clear_gimmick() -> void:
    gimmick_type = GimmickType.NONE
    gimmick_state = 0
    gimmick_durability = 0
    gimmick_data = {}
```

### 수인 기준

- [ ] `cell.gd`에 `gimmick_type`, `gimmick_state`, `gimmick_durability`, `gimmick_data` 필드가 추가된다
- [ ] `has_gimmick()`, `set_gimmick()`, `clear_gimmick()` 유틸리티가 동작한다
- [ ] 기존 기믹 없는 셀(`NONE`)에서 기존 동작에 regression이 없다
- [ ] Grid 초기화(`grid.gd`)에서 기본값으로 `NONE` 설정 확인

## Task 3: 레벨 JSON 기믹 배치 — 스키마 확장 + LevelLoader 수정

### 설명

레벨 JSON 스키마에 기믹 배치 필드를 추가하고, `LevelLoader`가 이를 파싱하여 셀에 기믹을 설정하도록 수정한다.

### 상세 구현 노트

#### JSON 스키마 확장

기존 레벨 JSON에 `gimmicks` 배열을 추가한다:

```json
{
  "grid_size": 7,
  "colors": 4,
  "turns": 25,
  "target": 30,
  "star_thresholds": [5, 10, 15],
  "initial_grid": [[1,2,3,...], ...],
  "gimmicks": [
    {
      "x": 3,
      "y": 2,
      "type": "locked",
      "durability": 1
    },
    {
      "x": 0,
      "y": 0,
      "type": "stone"
    },
    {
      "x": 5,
      "y": 5,
      "type": "ice",
      "durability": 2
    },
    {
      "x": 4,
      "y": 3,
      "type": "rainbow"
    },
    {
      "x": 2,
      "y": 1,
      "type": "paint_bucket",
      "data": { "direction": "row" }
    },
    {
      "x": 6,
      "y": 6,
      "type": "time",
      "data": { "bonus_seconds": 5.0 }
    }
  ]
}
```

#### type 문자열 ↔ GimmickType 매핑

`LevelLoader`에 문자열-enum 변환 딕셔너리를 둔다:

```gdscript
const TYPE_MAP: Dictionary = {
    "locked": GimmickType.LOCKED,
    "stone": GimmickType.STONE,
    "ice": GimmickType.ICE,
    "rainbow": GimmickType.RAINBOW,
    "anchor": GimmickType.ANCHOR,
    "paint_bucket": GimmickType.PAINT_BUCKET,
    "coin": GimmickType.COIN,
    "chain_mult": GimmickType.CHAIN_MULT,
    "star": GimmickType.STAR,
    "spread": GimmickType.SPREAD,
    "fade": GimmickType.FADE,
    "time": GimmickType.TIME,
    "poison": GimmickType.POISON,
}
```

#### LevelLoader 수정 사항

1. `gimmicks` 배열 파싱 → 각 항목에서 (x, y, type, durability?, data?) 추출
2. Grid 초기화 후 해당 좌표의 Cell에 `set_gimmick()` 호출
3. 유효성 검증:
   - 좌표가 그리드 범위 내인지
   - type 문자열이 TYPE_MAP에 존재하는지
   - 돌 칸(`stone`)이면 셀 색상을 -1(무색)으로 설정
   - 모드 전용 기믹 검증: 스토리 모드에서 S4/S5 기믹이 있으면 경고 로그, 무한 모드에서 S1/S2/S3 기믹이 있으면 경고 로그

#### 돌 칸 특수 처리

`stone` 기믹이 배치된 셀은 색상이 없어야 한다. `initial_grid`의 해당 좌표 색상값은 무시하고 `-1`(무색)로 강제 설정한다.

### 수인 기준

- [ ] `gimmicks` 필드가 포함된 레벨 JSON을 `LevelLoader`가 정상 파싱한다
- [ ] `gimmicks` 필드가 없는 기존 레벨 JSON도 하위 호환으로 정상 로드된다 (빈 배열 기본값)
- [ ] 파싱된 기믹 정보가 Cell의 기믹 슬롯에 올바르게 설정된다
- [ ] 범위 밖 좌표, 알 수 없는 타입 등 오류 시 `push_warning()` 출력 후 해당 항목 스킵
- [ ] stone 기믹 셀의 색상이 -1로 설정된다

## Task 4: M1 잠긴 칸 (Locked) 구현

### 설명

잠긴 칸은 색 변경이 불가하며, 오직 라인(행/열) 완성 파괴를 통해서만 제거할 수 있다. 잠긴 칸은 고유 색상을 가지고 있으며 BFS 탐색에는 포함되지만 리컬러되지 않는다.

### 상세 구현 노트

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `false` | 리컬러 차단 |
| `can_bfs_traverse()` | `true` | BFS 탐색 진입 허용 (동색이면 그룹에 포함되지만 색은 안 바뀜) |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 가능 (라인 완성 경유) |
| `on_destroy()` | `{destroyed: true}` | 기믹 제거, 셀 비우기 |
| `on_gravity()` | `true` | 정상 낙하 |

#### BFS 리컬러 로직 수정 (`flood_fill.gd`)

BFS로 연결 그룹을 찾은 후, 실제 리컬러 적용 단계에서 각 셀의 `can_recolor()`를 체크한다:

```
1. BFS로 동색 연결 그룹 탐색 (잠긴 칸도 동색이면 그룹에 포함)
2. 그룹 내 각 셀에 대해:
   - handler = GimmickRegistry.get_handler(cell.gimmick_type)
   - if handler.can_recolor(cell, new_color):
       cell.color = new_color  # 리컬러 적용
   - else:
       pass  # 잠긴 칸은 색 유지
```

> **중요**: 잠긴 칸은 BFS 그룹에는 포함되지만 리컬러 대상에서 제외된다. 이는 잠긴 칸 너머의 같은 색 셀도 그룹에 포함될 수 있음을 의미한다.

#### 라인 파괴와의 관계

잠긴 칸도 라인(행/열) 완성 판정에 참여한다. 행의 모든 셀이 같은 색이면 (잠긴 칸 포함) 라인이 완성되고, 잠긴 칸도 함께 파괴된다.

#### 엣지 케이스

1. **잠긴 칸만으로 이루어진 행**: 모두 같은 색이면 라인 완성 → 전부 파괴. 리컬러로는 어떤 칸도 바꿀 수 없으므로, 다른 행의 중력 낙하로 색이 맞아야 한다.
2. **잠긴 칸 + 무지개 칸 동일 행**: 무지개는 모든 색 매칭이므로 행 완성 판정 시 무지개를 와일드카드로 취급 → 나머지가 단색이면 완성.
3. **잠긴 칸 터치 시**: 사용자가 잠긴 칸을 직접 터치하면 BFS 그룹에 포함되지만 색이 안 바뀌므로 해당 셀 색상은 유지. 나머지 비잠금 셀만 리컬러.

#### 비주얼

- 셀 위에 자물쇠 아이콘 오버레이
- 구현: `_draw()` 또는 `Control` 오버레이로 잠금 아이콘 도형 그리기 (원 + 직사각형 조합)
- 색상: 셀 색상 위에 반투명 회색 자물쇠

### 수인 기준

- [ ] 잠긴 칸을 터치/BFS 리컬러해도 색상이 변경되지 않는다
- [ ] 잠긴 칸이 포함된 행/열이 단일 색으로 완성되면 잠긴 칸도 함께 파괴된다
- [ ] 잠긴 칸 너머 같은 색 셀이 BFS 그룹에 포함된다 (잠긴 칸이 BFS를 차단하지 않음)
- [ ] 잠긴 칸 위에 자물쇠 오버레이가 표시된다
- [ ] 파괴 후 기믹 슬롯이 NONE으로 초기화된다

- `scripts/core/flood_fill.gd` (수정 필요)
- `scripts/core/row_destroy.gd` (수정 필요)

## Task 5: M2 돌 칸 (Stone) 구현

### 설명

돌 칸은 색상이 없고, BFS 연결을 차단하며, 어떤 방법으로도 파괴할 수 없다. 그리드의 지형 장애물 역할을 한다.

### 상세 구현 노트

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `false` | 리컬러 불가 |
| `can_bfs_traverse()` | `false` | BFS 진입 차단 — 돌 너머 셀은 별도 그룹 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `false` | 파괴 불가 |
| `on_destroy()` | `{destroyed: false}` | 절대 파괴되지 않음 |
| `on_gravity()` | `false` | 낙하하지 않음 (고정 위치) |

#### BFS 탐색 수정 (`flood_fill.gd`)

BFS 탐색 시 인접 셀 큐 추가 전에 `can_bfs_traverse()`를 체크한다:

```
for neighbor in get_neighbors(cell):
    handler = GimmickRegistry.get_handler(neighbor.gimmick_type)
    if not handler.can_bfs_traverse(neighbor, cell):
        continue  # 돌 칸이면 탐색 차단
    if neighbor.color == target_color or handler.is_bfs_wildcard(neighbor):
        queue.append(neighbor)
```

#### 라인 완성 판정에서의 돌 칸

돌 칸은 색상이 없으므로(-1) 행/열 단일 색 판정에서 "불일치"로 처리된다. 따라서:
- 돌 칸이 포함된 행/열은 **절대 라인 완성되지 않는다**.
- 이는 레벨 디자인에서 "파괴 불가능한 행"을 만드는 데 활용된다.

#### 중력과 돌 칸

돌 칸은 중력 낙하하지 않는다(`on_gravity() → false`). 돌 칸 위의 빈 공간은 돌 칸 위쪽 셀들끼리 압축된다. 돌 칸 아래 빈 공간은 돌 칸 아래쪽 셀들끼리 압축 후 버퍼에서 충전된다.

> **중요**: 돌 칸은 해당 열을 상하 두 영역으로 분할하는 효과를 가진다. `gravity.gd`에서 열 처리 시 돌 칸을 경계로 세그먼트를 나누어 각각 독립적으로 압축해야 한다.

```
예시 (5×1 열):
  [빈칸]    → [  A  ]     ← 버퍼에서 충전
  [  A  ]    → [  B  ]
  [돌 칸]    → [돌 칸]     ← 고정
  [빈칸]    → [  C  ]     ← 아래 세그먼트 압축
  [  C  ]    → [새칸]     ← 버퍼에서 충전
```

#### 엣지 케이스

1. **돌 칸으로 둘러싸인 셀**: BFS가 돌 칸에 의해 완전히 격리된 셀은 해당 셀만 단독 그룹이 된다.
2. **돌 칸 + 페인트통 (S07)**: 페인트통이 행/열 변환 시 돌 칸은 영향받지 않는다 (can_recolor=false).
3. **초기 배치**: 돌 칸 좌표의 `initial_grid` 색상값은 무시, -1로 강제.
4. **연속 돌 칸**: 한 열에 돌 칸이 여러 개 있으면 열이 3개 이상 세그먼트로 분할된다.

#### 비주얼

- 셀 전체를 회색(#808080) 솔리드 블록으로 렌더링
- 약간의 테두리 음영으로 입체감 부여
- 색상 표시 없음 (기존 셀 색상 렌더링 생략)
- 구현: `_draw()`에서 `draw_rect()` + 약간 어두운 테두리

### 수인 기준

- [ ] BFS 탐색이 돌 칸에서 완전히 차단된다
- [ ] 돌 칸이 포함된 행/열은 라인 완성 판정에서 항상 실패한다
- [ ] 어떤 파괴 메커니즘으로도 돌 칸이 제거되지 않는다
- [ ] 중력 처리 시 돌 칸은 고정되며, 열이 돌 칸 기준으로 세그먼트 분할 압축된다

## Task 6: M3 얼음 칸 (Ice) 구현

### 설명

얼음 칸은 라인 파괴 2회에 걸쳐 제거된다. 1회차에 금(crack)이 가고, 2회차에 완전 파괴된다. 얼음은 기존 셀 위에 오버레이로 존재하며, 얼음 아래의 셀은 정상적으로 색상을 가지고 리컬러도 가능하다.

### 상세 구현 노트

#### 핵심 설계: 얼음은 "보호막"이다

- 얼음은 셀의 색상/BFS 동작에 영향을 주지 않는다
- 셀은 정상적으로 리컬러 가능
- 얼음의 효과: 라인 파괴 시 **셀이 즉시 비워지지 않고** 얼음 내구도만 감소
- 내구도(durability) = 2 → 1회 파괴 시 1로 감소 (crack 상태), 2회 파괴 시 0 → 셀 파괴

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 (얼음은 보호막일 뿐) |
| `can_bfs_traverse()` | `true` | BFS 탐색 정상 통과 |
| `is_bfs_wildcard()` | `false` | 와일드카드 아님 |
| `can_destroy()` | `true` | 파괴 대상에 포함 가능 |
| `on_destroy()` | durability > 1: `{destroyed: false}`, durability 1→0: `{destroyed: true}` | 내구도 감소 처리 |
| `on_gravity()` | `true` | 정상 낙하 (얼음 상태 유지한 채 낙하) |

#### `on_destroy()` 상세 로직

```gdscript
func on_destroy(cell) -> Dictionary:
    cell.gimmick_durability -= 1
    if cell.gimmick_durability <= 0:
        cell.clear_gimmick()
        return {destroyed: true}   # 셀이 실제로 비워짐
    else:
        cell.gimmick_state = 1     # 1 = cracked
        return {destroyed: false}  # 셀은 유지됨 (색상 보존)
```

#### `row_destroy.gd` 수정

라인 완성 시 파괴 대상 셀 처리 로직에서:

```
for cell in line_cells:
    handler = GimmickRegistry.get_handler(cell.gimmick_type)
    if not handler.can_destroy(cell):
        continue  # 돌 칸 등
    var result = handler.on_destroy(cell)
    if result.destroyed:
        destroy_set.add(cell)       # 실제 비우기 대상
    else:
        protected_set.add(cell)     # 얼음 보호로 유지된 셀
```

#### 라인 완성 판정과 얼음

얼음 칸은 색상을 가지고 있으므로 라인 완성 판정에 정상 참여한다. 모든 셀이 동색이면 라인이 완성되고, 얼음 셀은 `on_destroy()`가 호출되어 내구도 감소만 일어난다.

#### 연쇄(chain) 상호작용

- 1회차 파괴로 얼음에 금이 간 후, 같은 연쇄 과정에서 중력→재배치로 다시 라인이 완성되면 **2회차 파괴가 적용**되어 얼음이 깨질 수 있다.
- 한 턴 내 연쇄 콤보에서 얼음이 2단계에 걸쳐 파괴되는 것은 정상 동작이다.

#### 엣지 케이스

1. **얼음 + 잠긴 칸**: 한 셀에 기믹은 1개만 가능하므로 동시 적용 불가. 레벨 디자인으로 방지.
2. **얼음 셀의 BFS 그룹 파괴**: 라인 완성에 의한 파괴에서만 얼음이 의미 있다. BFS 그룹 파괴(라인 완성에 연결된 같은 색 그룹)에도 얼음 보호가 적용된다.
3. **얼음 crack 후 리컬러**: crack 상태의 얼음 셀도 정상 리컬러 가능. 내구도는 유지.

#### 비주얼

- **정상 상태 (durability=2)**: 셀 색상 위에 반투명 하늘색(#88CCFF, alpha 0.4) 오버레이
- **crack 상태 (durability=1)**: 동일 오버레이 + 대각선 금 라인(흰색/밝은 회색)
- 구현: `_draw()`에서 `draw_rect()` (반투명 tint) + crack 시 `draw_line()` 2~3개

### 수인 기준

- [ ] 얼음 칸이 하늘색 오버레이로 표시된다
- [ ] 얼음 칸의 셀이 정상적으로 리컬러된다
- [ ] 1회 라인 파괴 시 crack 상태로 전환되며 금 비주얼이 표시된다
- [ ] 2회 라인 파괴 시 얼음이 제거되고 셀이 파괴(비워짐)된다
- [ ] 한 턴 내 연쇄로 2단계 파괴가 가능하다
- [ ] 얼음 셀이 중력 낙하 시 얼음 상태를 유지한 채 이동한다

## Task 7: M4 무지개 칸 (Rainbow) 구현

### 설명

무지개 칸은 와일드카드로, BFS 탐색 시 어떤 색과도 연결된다. 리컬러 시 해당 색으로 실제 변경된다. 무지개 칸이 리컬러되면 무지개 기믹은 제거되고 일반 셀이 된다.

### 상세 구현 노트

#### 핵심 설계

- 무지개 칸은 자체 색상을 가지지 않는다 (또는 특수값 -2 등으로 표현)
- BFS에서 인접 셀의 색상과 비교할 때, 무지개 칸은 항상 "같은 색"으로 판정
- 리컬러 시 무지개 기믹이 소모되어 일반 색상 셀로 변환

#### 색상 처리 방안

**방안 A (권장)**: 무지개 칸의 color = -2 (RAINBOW_COLOR 상수). 렌더링 시 특수 처리.

```gdscript
const RAINBOW_COLOR: int = -2
```

#### 라이프사이클 훅 구현

| 훅 | 반환값 | 설명 |
|---|--------|------|
| `can_recolor()` | `true` | 리컬러 가능 |
| `can_bfs_traverse()` | `true` | BFS 탐색 통과 |
| `is_bfs_wildcard()` | `true` | **모든 색과 매칭** |
| `can_destroy()` | `true` | 파괴 가능 |
| `on_destroy()` | `{destroyed: true}` | 일반 파괴 |
| `on_gravity()` | `true` | 정상 낙하 |
| `on_recolor()` | 기믹 제거 | 리컬러되면 일반 셀로 전환 |

#### BFS 탐색 수정 (상세)

```
# flood_fill.gd 수정
func _colors_match(cell_a, cell_b) -> bool:
    var handler_a = GimmickRegistry.get_handler(cell_a.gimmick_type)
    var handler_b = GimmickRegistry.get_handler(cell_b.gimmick_type)
    if handler_a.is_bfs_wildcard(cell_a) or handler_b.is_bfs_wildcard(cell_b):
        return true
    return cell_a.color == cell_b.color
```

> **주의**: 무지개 칸 두 개가 인접하면 서로 매칭되어 하나의 BFS 그룹이 된다.

#### `on_recolor()` 처리

```gdscript
func on_recolor(cell, old_color, new_color) -> void:
    cell.color = new_color
    cell.clear_gimmick()  # 무지개 기믹 소모 → 일반 셀로 전환
```

#### 라인 완성 판정에서의 무지개

무지개 칸(color=-2)은 행/열 단일 색 판정 시 **와일드카드**로 취급한다:
- 행의 셀 색상이 `{3, 3, -2, 3, 3, 3, 3}`이면 → 무지개를 제외한 나머지가 모두 3이므로 라인 완성
- 행의 셀 색상이 `{3, 2, -2, 3, 3, 3, 3}`이면 → 3과 2가 혼재하므로 라인 미완성
- 행 전체가 무지개면 → 라인 완성 (와일드카드끼리는 매칭)

`row_destroy.gd` 라인 완성 판정 로직 수정:

```
func is_line_complete(cells: Array) -> bool:
    var base_color = -1
    for cell in cells:
        if cell.gimmick_type == GimmickType.STONE:
            return false  # 돌 칸 포함 → 절대 완성 불가
        handler = GimmickRegistry.get_handler(cell.gimmick_type)
        if handler.is_bfs_wildcard(cell):
            continue  # 무지개는 스킵
        if base_color == -1:
            base_color = cell.color
        elif cell.color != base_color:
            return false
    return base_color != -1 or all_are_wildcard  # 모두 무지개여도 완성
```

#### 엣지 케이스

1. **무지개 칸 터치 시 BFS**: 무지개 칸을 직접 터치하면 인접한 **모든** 색의 셀과 연결된다. 이로 인해 거대한 BFS 그룹이 형성될 수 있다.
2. **무지개 + 잠긴**: 한 셀에 동시 적용 불가 (기믹 1개 제한).
3. **무지개 리컬러 후 라인 체크**: 무지개가 리컬러되어 일반 셀로 변환된 후 라인 완성 판정이 이루어지므로, 무지개 소모 후에도 라인이 완성될 수 있다.
4. **무지개 + 무지개 인접**: 두 무지개가 인접하면 서로 연결. 무지개 체인으로 먼 셀들이 하나의 BFS 그룹에 포함될 수 있다.
5. **무지개 터치 시 색 결정**: 사용자가 무지개 칸을 터치하면 컬러 큐의 현재 색이 그 무지개 셀에 적용되고, BFS로 연결된 모든 셀(다양한 색)이 해당 색으로 리컬러된다. 이때 `can_recolor()`가 false인 셀(잠긴 칸)은 리컬러에서 제외.

#### 비주얼

- 무지개 그라디언트: 여러 색상이 가로 줄무늬 또는 사선 그라디언트로 표시
- 구현: `_draw()`에서 6개 컬러 밴드를 `draw_rect()`로 가로 분할 렌더링
- 또는 Shader: 간단한 fragment shader로 UV 기반 무지개 그라디언트

### 수인 기준

- [ ] 무지개 칸이 무지개 그라디언트로 표시된다
- [ ] BFS 탐색 시 무지개 칸이 어떤 색 인접 셀과도 연결된다
- [ ] 무지개 칸 리컬러 시 지정 색으로 변경되고 기믹이 제거되어 일반 셀이 된다
- [ ] 라인 완성 판정 시 무지개 칸이 와일드카드로 취급된다
- [ ] 무지개 칸 두 개가 인접하면 서로 BFS 연결된다
- [ ] 무지개 칸을 직접 터치 시 인접한 모든 색 셀과 거대 그룹이 형성된다

## 코어 시스템 기믹 훅 통합 가이드

> 이 섹션은 기존 코어 스크립트에 기믹 훅을 어떻게 삽입해야 하는지 종합 가이드이다.

### `flood_fill.gd` 수정 가이드

```
find_connected_group(grid, start_x, start_y, target_color) -> Array:
    var group = []
    var queue = [grid.get_cell(start_x, start_y)]
    var visited = {}

    while queue not empty:
        cell = queue.pop_front()
        if cell in visited: continue
        visited[cell] = true

        # 기믹 핸들러 조회
        var handler = GimmickRegistry.get_handler(cell.gimmick_type)

        # 색 매칭 판정 (와일드카드 고려)
        if not _colors_match(cell, target_color, handler):
            continue

        group.append(cell)

        for neighbor in get_4_neighbors(cell):
            var n_handler = GimmickRegistry.get_handler(neighbor.gimmick_type)
            # BFS 진입 가능 여부
            if not n_handler.can_bfs_traverse(neighbor, cell):
                continue
            queue.append(neighbor)

    return group

_colors_match(cell, target_color, handler) -> bool:
    if handler.is_bfs_wildcard(cell):
        return true
    return cell.color == target_color
```

### `row_destroy.gd` 수정 가이드

```
check_line_completion(cells: Array) -> bool:
    var base_color = -1
    var all_wildcard = true

    for cell in cells:
        var handler = GimmickRegistry.get_handler(cell.gimmick_type)
        if not handler.can_destroy(cell):
            return false  # 돌 칸 등 파괴 불가 셀 포함 → 완성 불가
        if handler.is_bfs_wildcard(cell):
            continue  # 무지개는 스킵
        all_wildcard = false
        if base_color == -1:
            base_color = cell.color
        elif cell.color != base_color:
            return false

    return base_color != -1 or all_wildcard

process_line_destruction(cells: Array) -> Dictionary:
    var destroyed = []
    var protected = []

    for cell in cells:
        var handler = GimmickRegistry.get_handler(cell.gimmick_type)
        if not handler.can_destroy(cell):
            continue
        var result = handler.on_destroy(cell)
        if result.destroyed:
            destroyed.append(cell)
        else:
            protected.append(cell)

    return {destroyed = destroyed, protected = protected}
```
