## 1. Overview

### 1.1 Purpose

7x7 그리드 + 7행 버퍼 구조에서 컬러 큐에서 배정된 색으로 셀을 터치하여 BFS 연결 그룹을 리컬러하고, 행 또는 열이 단일 색으로 완성되면 해당 라인과 연결 그룹을 파괴, 중력 낙하 후 연쇄 콤보가 이어지는 퍼즐 게임을 구현한다. 스토리 모드(턴 제한, N블록 파괴 목표)와 무한 모드(시간 제한, 대량 파괴 시 보너스 시간)의 이중 구조로 리텐션을 유지한다.

- HTML 프로토타입 참조: `pictory-puzzle-prototype_2.html`

### 1.2 Background

- 니노쿠니 IP의 후냐와 이마젠 캐릭터를 활용한 캐주얼 퍼즐 게임
- 기존 컬러 플러드 게임(Flood-It 등)의 "모든 칸 통일" 방식에서 탈피 → 라인(행/열) 파괴 + 중력 + 연쇄 콤보로 차별화
- 동료 이마젠 스킬 시스템 + 스토리 통합으로 단순 퍼즐 이상의 경험 제공
- 모바일 우선 설계

### 1.3 Related Documents

- GDD: `docs/game-design-document.md`
- HTML Prototype: `pictory-puzzle-prototype_2.html`

## 2. Scope

### 2.1 In Scope

- [ ] 핵심 퍼즐 메커니즘 (7x7 그리드 + 7행 버퍼 + BFS 리컬러 + 라인(행/열) 파괴 + 중력 낙하 + 연쇄 콤보)

- [ ] 스토리 대화 시스템 (캐릭터 일러스트 + 텍스트)

- [ ] 밸런싱 시뮬레이터 (다중 AI 전략 시뮬레이션)
- [ ] 레벨 자동 생성기
- [ ] 모바일 터치 UI 최적화
- [ ] 동료 이마젠 시스템 (스토리 클리어 시 동료 합류 + 고유 스킬)
- [ ] 이마젠 편성 및 스킬 발동 (쿨타임 기반, 턴 미소모)

## 4. Success Criteria

### 4.1 Definition of Done

- [ ] 모든 FR-01~FR-10 (핵심 메커니즘) 구현 및 동작 확인
- [ ] 스토리 모드 챕터 1 (10스테이지) 완전 플레이 가능
- [ ] 무한 모드 5분 이상 연속 플레이 가능
- [ ] 동료 이마젠 최소 2마리 획득 및 스킬 사용 가능
- [ ] 컬렉션 UI에서 후냐 커스터마이징/이마젠 편성/플레이어 아이콘 확인 가능
- [ ] 플레이어 프로필 (닉네임 + 아이콘) 메인 메뉴에 표시
- [ ] 밸런싱 시뮬레이터 구동 확인
- [ ] 모바일 빌드 (Android) 실기기 테스트 통과

### 4.2 Quality Criteria

- [ ] 60fps 유지 (타겟 디바이스)
- [ ] 연쇄 콤보 처리 시 프레임 드롭 없음
- [ ] 메모리 릭 없음 (장시간 무한 모드 플레이)
- [ ] 앱 크래시율 < 0.1%

## 5. Risks and Mitigation

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| 연쇄 콤보 무한 루프 | High | Low | 최대 연쇄 횟수 제한 (예: 20회). 루프 탐지 로직 추가 |
| 기믹 시스템 재설계 지연 | High | Medium | 기믹 없이 코어 메커니즘 우선 완성. 기믹은 별도 Phase로 분리하여 점진 도입 |
| 컬러 큐 밸런싱 어려움 | Medium | Medium | 스트라이드/오프셋 파라미터를 Inspector에 노출하여 빠른 조정. 시뮬레이터로 검증 |
| 중력 + 라인 파괴 애니메이션 타이밍 | Medium | Medium | 로직과 연출 분리. 로직은 즉시 처리, 연출은 큐잉하여 순차 재생 |
| 재화 그라인딩 피로도 | Medium | Medium | 시뮬레이터로 평균 획득량 검증. 해금 비용 적정성 밸런싱 |
| 니노쿠니 IP 저작권 문제 | High | Unknown | IP 사용 범위 및 라이선스 사전 확인 필요. 문제 시 오리지널 캐릭터로 대체 |
| Godot 모바일 성능 | Medium | Low | 프로토타입 단계에서 타겟 디바이스 성능 확인. 필요 시 렌더링 최적화 |
| 이마젠 스킬 밸런싱 | Medium | Medium | 스킬이 퍼즐 난이도를 무력화하지 않도록 쿨타임/효과 범위 조절. 별 평가와 분리 |
| 컬렉션 드롭율 밸런싱 | Low | Medium | 시뮬레이터로 기대 수집 기간 검증 |

## 6. Architecture Considerations

### 6.1 기술 스택

| Decision | Selected | Rationale |
|----------|----------|-----------|
| Engine | Godot 4.x | 오픈소스, 모바일 지원, GDScript 생산성 |
| Language | GDScript | Godot 네이티브, 빠른 프로토타이핑 |
| 레벨 데이터 | JSON | 가독성, 에디터 호환, 직렬화 용이 |
| 시뮬레이터 | GDScript 또는 별도 Python 스크립트 | 성능 필요시 Python으로 분리 |
| 플랫폼 | Android (우선) → iOS | 모바일 우선 |
| 프로토타입 | HTML/JS | `pictory-puzzle-prototype_2.html`로 코어 메커니즘 검증 완료 |

### 6.3 핵심 아키텍처 패턴

- **데이터-뷰 분리**: 그리드 데이터(grid.gd)와 렌더링(씬)을 분리. 시뮬레이터는 데이터 레이어만 사용.
- **코어 루프 파이프라인**: 리컬러 → 라인(행/열) 판정 → 파괴 → 중력 → 연쇄 재판정의 단계별 파이프라인 구조.
- **레벨 데이터 주도**: 모든 스테이지 정보는 JSON으로 외부화. 코드 변경 없이 레벨 추가/수정 가능.
- **모드 추상화**: `infinity_mode.gd`와 `story_mode.gd`가 공통 인터페이스를 공유. 게임 씬은 모드에 무관하게 동작.
- **컬러 큐 파라미터화**: 스트라이드 및 오프셋 파라미터를 Inspector에 노출하여 디자이너가 직접 조정 가능.
